## 摘要
视频在时间上存在冗余：相邻帧通常共享大部分物体、背景和布局。然而，现有的视频多模态大语言模型（视频MLLM）通常将每个采样帧编码为独立的RGB图像，导致视觉标记重复先前帧中已有的内容。因此，提出一种更直接的视频接口：仅在无法通过先前上下文很好地预测场景时发送完整的参考帧，否则传输帧间变化的紧凑描述。我们称这种接口为“预测性视觉编码”，并针对视频MLLM实例化为AdaCodec。只有在条件预测代价较高时，AdaCodec才会在参考帧上花费完整的视觉标记；否则，它将帧间变化（包括运动和预测残差）编码为紧凑的P-token。在所有11个基准测试中，在匹配的视觉标记预算下，AdaCodec相比Qwen3-VL-8B的逐帧RGB基线有所改进。即使在预算仅为1/7的情况下，使用32k标记的AdaCodec在所有长视频基准测试中均超过了224k基线；在五个通用视频基准测试上，它将平均分数提高了，同时将首次输出标记延迟从9.26秒大幅降低至1.62秒。

## Abstract
Video is temporally redundant: adjacent frames usually share most objects, background, and layout. Yet existing video multimodal large language models (video MLLMs) usually encode each sampled frame as an independent RGB image, causing visual tokens to repeat content already present in earlier frames. This suggests a more direct video interface: send a full reference frame only when the scene cannot be predicted well from prior context, and otherwise transmit a compact description of inter-frame changes. We call this interface a \emph{predictive visual code}, and instantiate it for video MLLMs as \textbf{AdaCodec}. AdaCodec spends full visual tokens on a reference frame only when its conditional predictive cost is high; otherwise, it encodes inter-frame changes, including motion and prediction residuals, as compact P-tokens. Across all eleven benchmarks, AdaCodec improves over the Qwen3-VL-8B per-frame RGB baseline at a matched visual-token budget. Even at $1/7$ the budget, AdaCodec with 32k tokens surpasses the 224k baseline on all long-video benchmarks; on five general-video benchmarks, it raises the average score while substantially cutting time-to-first-token from 9.26s to 1.62s.

## 1. 核心问题与研究动机

视频多模态大语言模型（视频MLLM）在处理长视频时面临严重的**时间冗余**问题：相邻帧通常共享大部分物体、背景和布局，但现有方法独立编码每帧为RGB图像，导致视觉token大量重复，造成：
- **覆盖-细节困境**：稀疏采样会遗漏短时事件，密集采样则快速消耗上下文窗口，增加延迟。
- **线性增长**：token数随帧数线性增长，有限上下文下难以同时保证覆盖范围和细节。

现有效率方法（帧选择、token压缩、记忆管理等）仍保留“独立RGB帧”的基本接口，未从根本上消除冗余。部分工作利用编码域信号（如运动向量），但往往将编码器输出视为固定信号，而非为MLLM推理优化编码本身。

**核心问题**：视频MLLM应消费什么样的视觉表示，才能既去除时间冗余，又保留推理所需证据？

## 2. 方法论

### 核心思想：预测性视觉编码

受生物视觉系统（编码预测误差而非原始信号）和现代视频编码（I帧/P帧结构）启发，**AdaCodec**将视频MLLM的视觉接口重新设计为预测性编码：
- **I帧**（参考帧）：仅当条件预测代价高时才发送完整ViT token。
- **P帧**（预测帧）：以紧凑的运动和残差token表示帧间变化。

### 关键技术细节

#### 2.1 MLLM导向的编码器重新设计

与面向回放的标准编码器不同，AdaCodec针对MLLM推理进行了四项核心重设计（表1）：

| 组件 | 回放编码器 | AdaCodec (MLLM导向) |
|------|-----------|---------------------|
| 块划分 | 可变块大小（优化比特率） | 16×16宏块与ViT patch网格对齐 |
| 运动参考 | 按编码语法选择参考帧 | 每P帧从前一采样帧估计（处理大时间间隔） |
| 搜索窗口 | 适配高帧率播放 | 放大局部窗口以适应低帧率下的大位移 |
| GOP调度 | 单独内容分析通路 | 复用运动搜索的预测成本，触发自适应I帧插入 |

#### 2.2 运动与残差编码

- 将视频划分为自适应GOP（图片组），每GOP第一帧为I帧，后续帧为P帧。
- 对每个宏块，在参考帧的局部窗口内搜索最佳匹配块，通过**SAD（绝对差之和）+ ℓ1运动正则**选择运动向量mbt：
  ```
  ℓbt(d) = SAD(xbt, x_{t-1}^{b+d}) + λ∥d∥₁
  mbt = argmin_{d∈Db} ℓbt(d)
  ```
- 残差rbt = 当前块 - 参考块；构建5通道张量ut = [rt, Mt]（残差+2通道运动场）。

#### 2.3 自适应GOP构建

- 帧级预测成本ℓt = Σ_b ℓbt(mbt)，当ℓt >阈值γ时插入新I帧（无需额外分析）。
- 阈值通过训练集调整，使中位GOP包含8个P帧；同时设置最大P帧数Kmax=16。

#### 2.4 双分支视觉token化架构

- **I帧编码器EI**：使用原始ViT（256个token/frame）。
- **P帧tokenizer EP**：将ViT patch嵌入从3通道扩至5通道（复制RGB权重、零初始化运动通道），附加可学习latent token（NP=16个），使用块注意力掩码防止patch token关注latent token。
- 适配Qwen3-VL的Conv3D stem（通过复制维度）、2×2空间合并、DeepStack注入路径。

#### 2.5 两阶段训练

**阶段1（教师特征对齐）**：
- 冻结教师视觉编码器，训练EP和轻量辅助预测器Hφ，用L1+余弦损失使预测特征逼近教师特征。
- 目标：学习紧凑P-token表示。

**阶段2（多模态对齐）**：
- 丢弃Hφ，保持EP冻结，通过标准自回归下一token预测损失（仅更新语言模型）对齐视觉编码与文本。

## 6. 主要结论与发现

1. **预测性编码优势**：即使在1/7 token预算下，AdaCodec在所有11个基准上维持甚至超越224k token基线；在匹配预算下**全面超越**每项基准。
2. **效率提升显著**：
   - token压缩84.6%（实际视频平均GOP长度10.21帧）。
   - TTFT从9.26s降至1.62s（含0.12s编码构建）。
   - E2EL从11.18s降至3.20s。
   - 峰值内存仅增加1.9GB。
3. **自适应GOP有效性**：缓慢变化场景（如异常检测）获得更长GOP，动态场景获得更多I帧刷新，提升了长视频任务性能（MLVU异常检测提升+20.6）。
4. **表示层必要性验证**：P-token优于全RGB、缩略图P甚至训练后的缩略图，证明**预测性编码表示本身**是关键，而非仅减少token数。
5. **编码设计验证**：ViT对齐的16×16宏块优于动态分区（+1.4~1.9），自适应GOP优于固定长度（+0.2~2.1）。

## 7. 优点

- **创新性**：将视频编码的预测性思想与MLLM视觉接口深度融合，直接重设计编码本身而非消费固定编码流，属于范式创新。
- **高效性**：实现真正的Pareto改进——更少token、更低延迟、更高精度。
- **鲁棒性**：跨11个基准、不同视频类型（长/短/动态/静态）均表现稳定。
- **消融充分**：从高层表示到底层编码参数全面验证，结论可靠性高。
- **实用性**：编码构建开销极低（0.12s/视频），易于集成到现有管道。

## 8. 不足与局限（论文自述+分析）

1. **固定分辨率**：未探索动态分辨率输入，限制了在复杂场景下的适应性。
2. **均匀P-token预算**：每P帧固定NP=16，未根据运动/残差复杂度动态调整，可能浪费或不足。
3. **未评估流式场景**：虽然因果I/P结构原则上支持流式，但无实证结果。
4. **计算成本考虑**：额外0.12s编码构建虽小，但在极端低延迟场景可能仍需优化，且依赖CPU。
5. **泛化边界**：仅基于Qwen3-VL-8B架构验证，对其他ViT/LLM架构的适配性未讨论。
6. **训练数据偏差**：使用公开数据集混合，可能偏向常见场景；未分析模型在罕见/极端视频上的行为。
7. **GOP阈值固定**：使用训练集固定的单一阈值，未自适应调整，可能在不同数据域间次优。
