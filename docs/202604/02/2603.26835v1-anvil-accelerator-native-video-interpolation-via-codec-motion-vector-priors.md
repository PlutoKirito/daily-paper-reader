## 摘要
移动端显示器的刷新率通常为 90-120 Hz，但大多数视频的编码帧率为 24-30 fps；实现实时的帧率翻倍要求在移动端神经网络处理单元（NPU）上，每个合成帧的处理时间需在 33.3 ms 以内。我们指出，主流的基于光流的视频插帧方法在移动加速器上面临三个结构性的部署障碍：空间采样算子超出了帧时间预算或缺乏硬件支持、迭代光流细化在 8 位训练后量化下失效，以及受内存限制（memory-bound）的算子主导了推理图。ANVIL 通过复用 H.264 解码器已计算出的运动矢量来预对齐输入帧，从而解决了这些障碍，并从加速器图中移除了学习型光流、空间采样和迭代累加。剩余的残差由一个以卷积为主的网络进行细化，该网络的推理图几乎完全由受计算限制（compute-bound）的算子组成。在骁龙 8 Gen 3 设备上，ANVIL 在 8 位整数精度下实现了 12.8 ms 的 1080p 网络推理；在 30 分钟的连续播放过程中，一个开源 Android 播放器在 54,623 个连续记录的样本中，每对插帧的端到端延迟中位数保持在 28.4 ms。逐算子因果分析表明，循环光流状态上的量化累加是导致迭代方法中整数量化失败的关键机制。目前的设计针对具有解码器暴露运动矢量的 H.264 播放场景。

## Abstract
Mobile displays refresh at 90-120 Hz, yet most video is encoded at 24-30 frames per second; real-time frame-rate doubling requires each synthesized frame within 33.3 ms on mobile neural processing units. We show that mainstream flow-based video frame interpolation faces three structural deployment barriers on mobile accelerators: spatial sampling operators exceed the frame budget or lack hardware support, iterative flow refinement collapses under 8-bit post-training quantization, and memory-bound operators dominate the inference graph. ANVIL addresses these barriers by reusing motion vectors already computed by the H.264 decoder to prealign input frames, removing learned optical flow, spatial sampling, and iterative accumulation from the accelerator graph. The remaining residual is refined by a convolution-dominated network whose inference graph is composed almost entirely of compute-bound operators. On a Snapdragon 8 Gen 3 device, ANVIL achieves 12.8 ms 1080p network inference in 8-bit integer precision; an open-source Android player sustains 28.4 ms median end-to-end latency per interpolated frame pair over 54,623 consecutively logged samples during 30-minute continuous playback. Per-operator causal analysis identifies quantized accumulation on recurrent flow states as a key mechanism behind integer quantization failure in iterative methods. The current design targets H.264 playback scenarios with decoder-exposed motion vectors.

作者指出主流基于光流的方法在移动 NPU 上存在三大障碍：
*   **算子不兼容：** 核心的 `grid_sample`（空间采样）算子在 NPU 上延迟极高或完全不支持。
*   **量化崩溃：** 移动 NPU 依赖 INT8 量化以保证速度，但迭代式光流细化在 INT8 下精度会严重崩溃。

### 2. 方法论：核心思想与技术细节
ANVIL 的核心思想是**“加速器原生设计”**，即通过改变算法结构来适配硬件，而非强行移植。
*   **三处理器协同流水线：**
    1.  **CPU（预处理）：** 从 H.264 解码器中提取现成的运动矢量（MV），进行零阶保持（ZOH）填充和 4× 下采样。
    2.  **GPU（预对齐）：** 使用 Vulkan 计算着色器对 MV 进行中值滤波和高斯平滑，随后进行双线性插值变形（Warp），生成预对齐的帧对。
    3.  **NPU（残差细化）：** 运行一个专门设计的轻量级 U-Net（UNet-v3b），仅预测预对齐帧与真实帧之间的残差。
*   **关键技术细节：**
    *   **算子约束：** 仅使用标准卷积（3x3）、ReLU、加法和转置卷积，避开 `grid_sample` 和注意力机制。
    *   **部署优化：** 在部署时将 Batch Normalization (BN) 融合进卷积权重；将 U-Net 的 Concat 连接改为 Add 连接，减少内存拷贝。
    *   **量化免疫：** 由于残差预测的动态范围小且无迭代累加，天然适配 INT8 量化。

### 6. 主要结论与发现
*   **量化失效机制：** 首次通过逐算子因果分析发现，迭代式方法中的 **“循环状态上的加法（Add-on-recurrent-state）”** 是导致 INT8 量化精度崩溃的主因。
*   **算子词汇表：** 证明了限制算子种类（仅用计算密集型算子）是实现移动端 1080p 实时性的唯一可行路径。

### 7. 优点
*   **极高的实用性：** 针对真实硬件约束设计，提供了开源的 Android 播放器实现。
*   **深度诊断：** 不仅提出了新方法，还深入分析了现有方法在 NPU 上失败的底层硬件原因。
*   **系统级优化：** 考虑了 CPU/GPU/NPU 的负载均衡和内存带宽瓶颈，而非单纯追求模型指标。

### 8. 不足与局限
*   **编解码器依赖：** 目前高度依赖 H.264 的运动矢量暴露，且需要软件解码（因为 Android MediaCodec 硬件解码 API 暂不开放 MV 导出）。

*   **场景限制：** 对于包含大量 B 帧或复杂编码结构的视频，MV 的可靠性会下降，导致插帧质量波动。
